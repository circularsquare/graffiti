# Graffiti ‚Äî Project Spec

## Concept

An interactive first-person 3D explorer of NYC's Financial District built on real city data, with the eventual goal of letting users walk (or fly) around and paint pixels on building surfaces ‚Äî like r/place, but on actual NYC buildings.

## Data Pipeline

```
NYC DoITT CityGML (EPSG:2263, feet)   √ó20 DA files
        ‚Üì
  scripts/build_tiles.py               (npm run build-tiles)
    calls convert_citygml.py per DA with --no-filter
    - reproject ‚Üí local meters
    - triangulate faces (earcut)
    - compute world-space AABB bounds
        ‚Üì
  public/tiles/cell_{gx}_{gz}.bin      one binary file per non-empty 100 m cell
  public/tiles/manifest.json           [ { gx, gz, bounds, cellEstimate }, ... ]
        ‚Üì
  src/TileManager.js
    - loads manifest on startup
    - fetches binary tile files as player approaches (within 200 m of AABB edge)
    - unloads tiles when player moves away (beyond ~275 m, sticky-margin)
    - delegates fetch + decode to src/tileWorker.js so the main thread
      is never blocked on network or parse work
        ‚Üì
  src/loadCityGML.js  ‚Üí  buildMeshesFromData()  ‚Üí  Three.js BufferGeometry
```

**Worker pipeline (src/tileWorker.js)** ‚Äî runs the entire tile-load pipeline
off the main thread, in **two phases** so the spawn gate doesn't wait on the
expensive seed scan:

- **Phase 1** ‚Äî mesh build: `fetch` ‚Üí `decodeTile` (binary, see format below) ‚Üí per-mesh Y-shift + bbox ‚Üí
  greedy near-coplanar **face extraction** (normal-dot > 0.9995, plane offset
  within 15 cm) ‚Üí **post-merge pass** that re-examines every face pair and
  merges any whose averaged normals + planes still agree (loosened to 30 cm
  for the post-merge ‚Äî greedy ordering can split coplanar triangles into
  separate faces when an intermediate triangle seeds its own) ‚Üí per-face
  face-tangent UVs ‚Üí flat normals ‚Üí per-vertex `lineCoord` for shader-drawn
  face borders ‚Üí roof + wall for each building are **merged into one MeshData**
  (`buildMergedMeshData`) so the renderer issues one draw call per building
  instead of two. Each face carries its `meshType` so main.js recovers it
  per-triangle via `faces[triFace[ti]].meshType`. Returns one MeshData per
  building (typed-array buffers ride the `postMessage` transfer list for
  zero-copy handoff). MeshData carries `triFace` (per-triangle face index),
  `faces: [{normal, planeD, meshType}, ‚Ä¶]`, and `buildingKeys:
  ['id:roof', 'id:wall']` listing which meshTypes the mesh covers.
- **Phase 2** ‚Äî seed scan: per-cell Sutherland-Hodgman triangle clip + random
  seed dice-roll, run against in-worker snapshots of position+uv. Cell
  identity uses the face's representative normal/planeD (not the triangle's)
  so all triangles in a face contribute to the same cells. After discovery,
  an **overlap dedupe pass** groups cells by `(buildingId, meshType, cu, cv)`
  and merges entries whose planes are within 30 cm and normals agree (~8¬∞)
  into a single anchor cellKey ‚Äî this prevents near-coplanar sibling faces
  from owning duplicate cells at the same visual position, which would
  otherwise paint over each other. Returns `CellBundle` per mesh (cellKeys
  + cellGeoms + seeds). Runs in the worker after phase 1's `postMessage`
  returns, so the main thread wraps + adds meshes to the scene in parallel
  with the scan.

On the main thread, `loadCityGML.js#wrapMeshData` allocates a `BufferGeometry`,
attaches the typed arrays, and uses the shared `BUILDING_MAT`.
`TileManager._applyCellDataToTile` matches each `CellBundle` to its wrapped
mesh by `buildingId` (one merged mesh per building) and stashes per-meshType
entries on `mesh.userData.cellDataByType[meshType]`, then fires
`onTileCellData`. `main.js#seedTileCells` iterates each mesh's `cellDataByType`,
copies cell Float32Arrays into `cellGeomCache`, calls `paintStore.seed()` for
cells in the seed list that aren't already painted, and calls
`rebuildBuildingPaint(srcMesh, buildingKey)` once per meshType. No triangle
iteration, no clipping on main ‚Äî the old bottleneck is gone.

If phase 2 arrives before the main thread has finished chunk-wrapping phase 1,
`TileManager._handleCellData` stashes the payload on `tile._pendingCellData`
and `_doLoad` applies it the moment `tile.meshes` is populated.

Protocol:
- main ‚Üí worker: `{ type:'load', tileId, file, seedConfig: { fraction, colors } }`
- worker ‚Üí main:
  - `{ type:'loaded',   tileId, meshes:   [MeshData,   ...] }` ‚Äî phase 1
  - `{ type:'cellData', tileId, cellData: [CellBundle, ...] }` ‚Äî phase 2
  - `{ type:'error',    tileId, error }`

**Binary tile format** (`cell_{gx}_{gz}.bin`, little-endian, encoded by
`build_tiles.py#encode_tile`, decoded by `tileWorker.js#decodeTile`):

```
Header (8 B):
  uint32 magic            'GFTI' = 0x49544647
  uint8  version          1
  uint8  reserved
  uint16 buildingCount

Per building (variable):
  uint16 roofTriCount
  uint16 wallTriCount
  uint8  idLength
  bytes  id (UTF-8)

[zero pad to 4-byte boundary]

Float32 vertex blob, one contiguous block (absolute world-space X, Y, Z):
  for each building, in order:
    roofTriCount * 9 floats
    wallTriCount * 9 floats
```

This is ~5√ó smaller than the previous JSON format (each float was ~19 bytes
of text vs. 4 bytes binary). Coords are absolute, not tile-local ‚Äî float32
ULP at NYC scale (~3 mm at 25 km from origin) is still 3√ó under the tightest
geometric tolerance. If coverage ever extends much further, revisit by
shifting X/Z to tile-local and positioning each tile's `THREE.Group` at the
cell origin (would also need to plumb the origin through the worker's cell
scan and `buildCellGeometry`, since both currently emit world-frame polygons
into the shared `paintGroup`).

If `public/tiles/manifest.json` is missing the app errors out on startup ‚Äî
there's no single-file fallback any more.

To regenerate all tiles:
```bash
pip install -r scripts/requirements.txt
npm run build-tiles                 # all 5 boroughs
npm run build-tiles -- --manhattan  # Manhattan-only subset (faster iteration)
```

### Landmark overrides

The citywide CityGML dataset has a handful of known-bad simplifications ‚Äî
e.g. Washington Square Arch ships as a solid block. `src/landmarks.js`
keeps a short config array of overrides, each of which lists
`hideBuildingIds` (the bad meshes to suppress) and a `buildTriangles()`
function returning local-space `{ roof: Float32Array, walls: Float32Array }`
for the replacement.

The replacement runs through the **same tileWorker pipeline as a regular
building** ‚Äî landmarks aren't a separate scene layer. At startup,
`prepareLandmarks({ terrain })` resolves each landmark's terrain Y via
`sampleAsync`, applies its `rotationY` + `position` + `terrainY` transform
to produce world-space triangles, and indexes the result by tile ID
(asserting each landmark fits in one 100 m cell ‚Äî bridges spanning many
cells aren't supported yet). When `TileManager` admits a tile, it calls
`resolveTileInjection(tileId)` to fetch any landmark payload and forwards
it in the worker's `load` message. Inside `tileWorker.js`, the `decodeTile`
result is filtered by `hideBuildingIds`, then `injectedBuildings` are
appended to the building list before face extraction begins. From there
landmarks get face merging, face-tangent UVs, `lineCoord`, the cell scan,
overlap dedupe, paint groups, and seeding identically to GML buildings.
Synthetic `buildingId = 'landmark_<name>'` keeps paint persistence stable
across sessions.

A `tilesWithLandmarks()` set lets `TileManager` defer admission of any
landmark-bearing tile until `prepareLandmarks` resolves (otherwise the
tile would render the bad mesh briefly before reload). Non-landmark tiles
load freely throughout.

To add a new override:
1. Aim the F3 crosshair at the bad building in-game.
2. Copy `bldg ...` and `hit (x, y, z)` from the debug HUD.
3. Add an entry to `LANDMARKS` in `src/landmarks.js`. Authoring shapes:
   procedural triangle emitters (see `buildWashingtonSquareArchTriangles`
   for a box-based example) work for simple geometry; richer shapes want
   Blender ‚Üí glTF ‚Üí triangle extraction (not wired yet).

### OSM overlay pipeline (streets, water, green)

```
OpenStreetMap (Overpass API, "highway"/"natural=water"/"leisure=park"/‚Ä¶)
        ‚Üì
  scripts/fetch_osm_features.py       (npm run build-osm)
    - derive bbox from public/tiles/manifest.json (¬±1 cell padding)
    - split the bbox into ~0.05¬∞ chunks and query each via Overpass
      (per-chunk cache under data/overpass_chunks/; resumes partial runs;
       drops the cache on --refresh). Tries kumi / overpass-api.de /
       overpass.fr mirrors in order on each chunk.
    - stitch relation member ways into closed outer/inner rings
    - earcut triangulate water / green polygons ‚Üí flat XZ triangle lists
    - **clip each source triangle and street polyline to every 125 m cell's
      bleed-expanded bbox** before bucketing (Sutherland-Hodgman for polygons,
      Liang-Barsky for polylines). This bounds per-tile payload at the tile
      scale ‚Äî without it, an ocean-sized source triangle from e.g. Upper NY
      Bay lived whole in every overlapping tile, and osmWorker._tessellateTriTops
      would iterate millions of 2 m terrain blocks across its full bbox at
      runtime (OOMing the tab at ~6 GB on hilly spawns). The clip is a ~7 %
      disk cost for ~1000√ó runtime memory savings.
    - encode each cell as a binary blob (see format below) and gzip it
        ‚Üì
  public/osm/cell_{gx}_{gz}.bin.gz    binary streets + water + green
  public/osm/manifest.json            [{id, bounds, streetCount}, ...]
        ‚Üì
  src/OsmManager.js
    - loads manifest on startup, tiles in/out within 300 m (unload at 400 m)
    - fetches each .bin.gz, pipes through DecompressionStream('gzip'), and
      decodes the binary layout back into the same { streets, water, green }
      shape downstream builders consume
    - in terrain mode, posts the decoded source to src/osmWorker.js: the
      worker clips every source triangle to the 2 m terrain-block grid and
      returns interleaved 7-float-per-vertex `topXz` buffers + the list of
      covered blocks (as transferables). Main then emits skirts against
      live terrain state, resolves per-vertex Y via `getBlockCorners`, and
      finalizes meshes. This keeps the ~500 ms tessellation off the main
      thread, which was the dominant per-tile-load spike.
    - per tile: a Three.js Group with up to three unlit meshes
      (water plane, green plane, street ribbons) at staggered Y offsets
      above the ground plane; no collision, no shadows
```

**Binary tile format** (`cell_{gx}_{gz}.bin.gz`, little-endian after gunzip,
encoded by `fetch_osm_features.py#encode_tile_binary`, decoded by
`OsmManager.js#_decodeOsmTile`):

```
Header (12 B):
  uint32 magic             'GOSM' = 0x4D534F47
  uint8  version           1
  uint8  reserved
  int16  gx
  int16  gz
  uint8  typeCount         # of distinct highway types used in this tile
  uint8  reserved

Type table (inline per tile):
  for each highway type:
    uint8  nameLen
    bytes  name (UTF-8)
  [pad to 4 B]

Streets section:
  uint32 streetCount
  per street:
    uint8  typeIdx         index into the type table above
    uint8  nameLen         0 = unnamed
    uint16 pointCount
    bytes  name (UTF-8)
    [pad to 4 B]
    int32  coords[pointCount * 2]    (tile-local decimetres, X then Z)

Water / Green sections (same shape, back-to-back):
  uint32 polyCount
  per poly:
    uint32 coordCount                (= triCount * 6; flat XZ tri list)
    int32  coords[coordCount]        (tile-local decimetres)
```

Coords are stored as int32 decimetres relative to each tile's world origin
`(gx * 125, gz * 125)`. int32 range (¬±214 km) accommodates huge ocean
triangles that extend far past a 125 m cell; 10 cm precision is finer than
the 0.24 m/px terrain canvas can resolve. After gzip, a representative 71 KB
JSON tile drops to ~11 KB ‚Äî the full OSM dataset ships roughly an order of
magnitude smaller than before.

Street ribbons are extruded from polylines in `_buildStreetMesh` using
per-`highway` widths (motorway ‚âà 11 m ‚Üí path ‚âà 2 m). Polygon features reuse
the pre-triangulated XZ lists from Python directly as `BufferGeometry.position`
attributes ‚Äî no runtime triangulation.

### Terrain pipeline (blocky DEM)

```
NYC 1 m DEM GeoTIFFs from USGS 3DEP (data/dem/*.tif, NAVD88 metres)
        ‚Üì
  scripts/fill_dem_nodata.py     (python scripts/fill_dem_nodata.py)
    - GDAL fillnodata on each DEM (100 m search radius) ‚Üí data/dem/*_filled.tif
    - one-shot preprocess; must run before bake_terrain. The fill is at the
      DEM level (not per-tile at runtime) so both tiles at a shared seam
      sample read identical values ‚Äî the invariant corner-smoothing relies
      on. Water / offshore tiles where fillnodata can't reach stay NODATA
      and clamp to WATER_LEVEL_Y at runtime (both sides agree, no seam leak).
        ‚Üì
  scripts/bake_terrain.py        (python scripts/bake_terrain.py)
    - reads data/dem/*_filled.tif (errors out if not present)
    - sample cell-centred 64√ó64 grid per 125 m tile (~2 m spacing) plus a
      1-cell perimeter pad for cross-tile corner smoothing
    - cell (gx, gz) addresses a 125 m square in GRID space, which is world XZ
      rotated by MANHATTAN_GRID_DEG (~29¬∞) so the blocky step walls align with
      Manhattan's avenues / streets. Sample positions are computed in grid
      space then rotated to world XZ before the CRS projection. Only terrain
      uses grid coords ‚Äî buildings, OSM, and player XZ are world-space
      throughout; geo.js#worldToGrid handles the boundary.
    - pick a per-tile int16 min (decimetres) and uint8 scale (decimetres per
      delta unit, ‚â•1) so the tile's range fits into 0..254; encode each
      sample as one uint8 delta (255 = NODATA)
    - gzip-compress the payload so application/octet-stream can ship compact
      without host-level compression support
        ‚Üì
  public/terrain/cell_{gx}_{gz}.bin   (v2: header{version,scale,res,pad,min} + uint8 deltas, gzipped)
  (no manifest ‚Äî deterministic URL; 404 = no terrain here)
        ‚Üì
  src/TerrainManager.js
    - streams cells within a fixed 400 m load radius (shared with OsmManager;
      independent of the adaptive building radius)
    - each of the 1024 samples per tile becomes a rectangular prism from a
      fixed BLOCK_BASE_Y (‚àí10 m) up to the sample's decoded elevation
    - within-tile side-face culling skips faces hidden behind a taller
      neighbour sample; cell-boundary sides are drawn conservatively
    - MeshLambertMaterial with an onBeforeCompile hook that (a) biases
      vertex normals toward world-up to flatten Lambert contrast and
      (b) injects the shared block-grid overlay (src/gridShader.js) ‚Äî
      analytic triplanar gridlines at BLOCK_SIZE (125/64 m) pitch that
      render on flat tops and vertical cliff sides alike. The same
      overlay is injected into OsmManager's draped materials, so the
      grid sits above OSM fills and below painted cells.
    - sample(x, z) returns the nearest block's top for collision/walking
    - sampleAsync(x, z) triggers a priority fetch (bypasses MAX_CONCURRENT_LOADS)
      for the containing cell and resolves to its sample ‚Äî used by teleport to
      know the spawn Y before moving the camera
```

The blocky renderer intentionally foregoes smoothing ‚Äî each DEM cell reads as
a discrete tile. OSM overlays (streets/water/green) are not composited onto
terrain; in terrain mode they're invisible (OsmManager still runs so
random-teleport can pick a street location). Street labels are suppressed
via the `showLabels: false` option on `OsmManager`.

Named streets get a flat label laid along their longest segment
(`_buildStreetLabels` / `_getLabelAsset`). Labels are built from the bundled
Helvetiker Bold typeface via `ShapeGeometry` (not a canvas texture), so text
stays crisp at any viewing distance. A per-name geometry is cached and shared
across tiles ‚Äî repeat names only add a mesh transform, not new geometry ‚Äî and
labels are dropped if the street segment is too short to fit them.

### Mobile controls

Touch-primary devices (`(hover: none) and (pointer: coarse)`) get a separate HUD, wired in `src/mobileControls.js` and gated by `IS_MOBILE` in `main.js`. There's no pointer lock, no map mode, no escape overlay. Walking and flying both work ‚Äî the player starts in walk mode and toggles fly via a dedicated button.

- **Drag anywhere on the canvas** to look (drag-the-world direction; YXZ euler, pitch clamped ¬±89¬∞).
- **Tap** the canvas (‚â§ 10 px movement, ‚â§ 220 ms) to paint the cell **under the finger** ‚Äî no crosshair on mobile. The tap's `(clientX, clientY)` is converted to NDC and passed through `paint.tryPaint(ndc)`; `hitCell(ndc)` raycasts from that point instead of screen-centre. Erase by selecting the erase swatch on the colorbar ‚Äî no separate erase gesture, no brush sweep.
- **Left joystick** (bottom-left): vertical = forward/back, horizontal = strafe. Dead zone 0.15. Throw magnitude scales speed from 1.15√ó (just past deadzone) to 1.75√ó (full throw) ‚Äî no sprint key on mobile, so the joystick doubles as the sprint input: push harder to go faster. Writes the same `keys.KeyW/A/S/D` flags physics already reads on desktop, so movement stays world-horizontal in fly mode (use the up/down buttons to change altitude).
- **Fly toggle** (above the joystick, wing icon, left-aligned with the joystick): taps `physics.toggleFly()` and shows an active highlight while flying. Also clears any held Space/Shift flags when exiting fly mode.
- **Up button** (top of the right column, always visible): walk mode tap = `physics.requestJump()`; fly mode hold = `keys.Space` ascend.
- **Down button** (below the up button, visible only in fly mode): hold to set `keys.ShiftLeft` descend.
- **Minimap** (165 px default, 297 px enlarged): double-tap teleports to that world point (same `fastTravelTo` used by desktop's M-mode) ‚Äî two taps within 300 ms and 30 px of each other, each tap single-finger and within 8 px slop; single-finger drag pans (same `adjustMinimapPan` math as desktop big-mode, heading-up-aware); two-finger pinch zooms via `adjustMinimapZoom(log2(ratio))`. ## Physics & Collision

- **Gravity**: Constant downward acceleration, capped at terminal velocity
- **Ground/roof detection**: Downward raycast for buildings + ground plane, plus a direct read of `terrainManager.sample(x, z)` as an extra floor candidate. The raycast origin sits at the foot, so when the player clips horizontally into a stepped-terrain block the ray starts inside the mesh and backface-culls to nothing; the heightmap read catches that case and returns the block top as a normal step-up surface. Both sources contribute ‚Äî highest wins.
- **Wall collision**: Per-axis raycasts at two heights (eye level + mid-torso) with wall-sliding. After movement, a push-out pass checks all 4 cardinal directions and ejects the player from any wall they've sunk into (e.g. after descending into geometry).
- **Fly mode**: Disables gravity; Space ascends (can clip through ceilings to escape buildings); Shift descends but is blocked by any surface below including the ground plane. Descent consults both the downward raycast and `terrainManager.sample()` so the player can't shove themselves through a step's top even when the ray origin is already inside the block. A post-movement rescue snaps the camera up to `terrainY + WALK_HEIGHT` whenever the feet end up inside a block ‚Äî but only when the required push is ‚â§ 1.5 m. Larger steps are treated as walls: the horizontal move is undone and the fly velocity zeroed on both horizontal axes, so cliffs block the player instead of teleporting them up the face. (If the player was already buried at the previous position ‚Äî e.g. a save restored inside a block ‚Äî the pop-up still fires regardless of magnitude so they can't get permanently stuck.) Q multiplies fly speed 3√ó. Movement velocity is exponentially smoothed (~100ms time constant) so starting and stopping in the air ease in and out instead of snapping.
- **Safe spawn**: On load, raycasts outward in 8 directions; walks outward in rings until a clear position is found

## Rendering

- **Engine**: Three.js (WebGL)
- **Shading**: `MeshToonMaterial` with a 3-step gradient map (hard-edged cel shading)
- **Grid**: Face-tangent UV projection (axes lie in the face plane) so the 1√ó1 m grid (`GRID_SIZE = 1.0`, exported from `loadCityGML.js`) is undistorted on walls, flat roofs, and sloped roofs alike. Canvas tile texture with blue grid lines on near-white background. Flat roofs use the building's dominant wall direction (longest horizontal edge in wall geometry) to orient the grid with the building footprint rather than world axes. UV axes are computed **per extracted face** (not per triangle), so the grid is continuous across triangulation diagonals even when the CityGML source's quads aren't quite coplanar.
- **Face outlines**: Drawn directly in the wall/roof fragment shader via `onBeforeCompile` injection. A per-vertex `lineCoord` attribute encodes the world-space perpendicular distance to each of the triangle's three edges; internal edges within a face are set to a large sentinel so only real face boundaries emit a line. T-junctions (two same-face polygons sharing a boundary where one side has an extra midpoint vertex the other doesn't) are caught by a fallback that scans for any same-face edge collinear within 2 cm and overlapping by ‚â• 1 cm. Thickness (`BORDER_HALF_WIDTH = 0.015 m`) matches the grid texture lines, and `fwidth()`-based screen-space anti-aliasing keeps lines crisp at any camera distance without subpixel flicker.
- **Buildings**: Separate roof + wall meshes, `DoubleSide`, `castShadow` only (no `receiveShadow` ‚Äî avoids shadow-acne on triangulated surfaces)
- **Lighting**: Directional sun (1.8) + ambient (0.35); `PCFSoftShadowMap`
- **Atmosphere**: Sky-blue background + exponential distance fog
- **FOV**: 100¬∞
- **Pigeons**: Grounded flocks stream in/out on a 150 m cell grid (`PigeonManager`, radius 200 m, unload margin 60 m) with a deterministic per-cell RNG so reloads don't reshuffle them. Each cell rolls 0‚Äì2 flock anchors (40 / 40 / 20 %); anchors that sit under a building (downward raycast against `buildingMeshes` shows a roof > 3 m above terrain) are rejected so flocks never spawn inside a wall. If buildings haven't loaded yet the tile defers to `_pendingTiles` and retries on the next tick. Each flock is 5‚Äì9 pigeons uniformly scattered in a 2.5 m disk around the anchor; each pigeon stands on its own per-point terrain sample with an independent peck-bob (`sin^8` of a 0.3‚Äì0.9 Hz phase ‚Äî mostly 0 with brief spikes). All active pigeons share one `InstancedMesh` of 1 m quads; vertex shader does a Y-axis-locked camera-facing billboard so sprites stay upright when you look down at them from flight, samples a single-frame standing-pigeon sprite (procedurally drawn on a canvas, nearest-filter), and honours scene fog. The quad is anchor-at-feet (shader shifts `position.y` by +0.5) so writing `py = terrain.sample(...)` plants the pigeon on the ground. No collision, no shadows.

## Paint System

Left-click while pointer-locked to paint the grid cell you're looking at (within 15 m). Right-click to erase. **Holding** either button turns it into a brush ‚Äî every new cell the crosshair sweeps across gets painted/erased ‚Äî and the whole hold commits as a single undo entry. `Ctrl/Cmd + Z` reverts the most recent stroke (up to 50 strokes kept in memory); `Ctrl/Cmd + Shift + Z` or `Ctrl + Y` redoes. Any new stroke clears the redo stack.

### Cell identity

Each paintable unit is identified by the key `buildingId:meshType:cellU:cellV:planeDKey`:

- `buildingId` ‚Äî from the CityGML source (`mesh.userData.buildingId`)
- `meshType` ‚Äî `'roof'` or `'wall'`
- `cellU`, `cellV` ‚Äî `Math.floor(intersection.uv)` from the Three.js raycast hit. The UV coordinates use the same face-tangent projection as the grid texture (1 UV unit = 1 m), so each integer step corresponds to one visible grid cell. Each face's UV origin is shifted so the face sits **centred** within its `ceil(width / GRID_SIZE)` cell span ‚Äî so the leftmost and rightmost cells end up the same width. A 3.4 m wide face yields cells of width 0.7 / 1 / 1 / 0.7 m instead of 1 / 1 / 1 / 0.4 m. Same cell count, but slivers are bigger and less likely to be unpaintable specks. A 0.7 m cylinder facet becomes a single cu=0 column.
- `planeDKey` ‚Äî `Math.round(planeD * 2)`, where `planeD` is the **face's** averaged plane offset (not the per-triangle one ‚Äî using triangle-local values would round into a different 50 cm bucket for any triangle whose normal drifted slightly from its face average). Bucketed to 50 cm so near-coplanar faces share a key, while parallel faces at different depths (e.g. a stepped facade) get distinct keys and can be painted independently.

`hitCell()` runs the tentative cellKey through `canonicalCellKey()`, which on a `cellGeomCache` miss scans the building's known cells for the same `(cu, cv)` and returns the nearest pdKey within 1 bucket (‚âà50 cm planeD). This redirects raycast hits onto the worker's overlap-dedupe anchor when the hit face's pdKey isn't the chosen one.

### Paint groups (sliver + narrow-face absorption)

Grid cells near building edges, on chamfered corners, or on cylindrical / many-facet structures can be very small ‚Äî every cell on a 24-facet cylinder is a narrow sliver, for example. To make these paintable as unified patches rather than fiddly per-click specks, cells with area < `GRID_SIZE¬≤ √ó 0.5` (~0.5 m¬≤) are grouped with adjacent cells at **tile build time** and painted together at runtime. The threshold is generous on purpose: it's not just true edge slivers but also "narrow but tall" cylinder-facet cells that benefit from being merged.

`tileWorker.buildCellGroups` produces the groups. Adjacency is by **shared world-space edge** (vertex pair within 1 cm) computed from each cell's post-dedupe clipped polygon, so groups cross face boundaries naturally. Slivers are processed smallest-first; each tries to join a neighbour's group (or create a new one) subject to four rules:

- **Length:** bbox on each axis of the anchor face's tangent frame ‚â§ `GRID_SIZE + 0.2` (1.2 m). Tight cap (20 cm slack over a normal cell) lets edge slivers pair with their adjacent full cell on the same face without chaining further.
- **Angle:** member face normal vs. anchor face normal ‚â• `cos 20¬∞`.
- **Longest-edge-share:** the shared edge between sliver S and candidate N must be ‚â• 90% of either cell's longest boundary edge. This bakes in "never merge two long slivers end-to-end" ‚Äî the short side touching at the ends is nobody's longest edge, so the rule rejects it. Two short slivers sharing their long side ‚Üí merges.
- **Cross-face restriction:** if sliver and group anchor are on different faces (different `face.normal` array reference ‚Äî all cells from one face share the face's normal object), the sliver's narrow dimension must be < 0.05 m (approximated as `area / longest-edge`). Cylinder facets stay singletons (each is already ~its own clean cell after per-face UV centring); thin chamfer slivers can still merge with adjacent walls. The restriction exists because we can't hide the building's underlying face-border line on cross-face merges, so we only do them when that line is short enough not to be visually distracting.

The worker ships `cellGroups: Array<Array<cellKey>>` in the phase-2 `cellData` message. Only groups of size ‚â• 2 are sent; singletons are implicit. On the main thread, every member of each group points to one shared `Set<cellKey>` in `cellGroups: Map<cellKey, Set<cellKey>>`, so `tryPaint` / `tryErase` is a single O(1) `.get()` followed by a batch. Groups are cleaned up alongside `cellGeomCache` in `TileManager._unload` via `cellGroupKeysByBuilding`.

After grouping, `buildCellGroups` runs a unified-offset pass: each grouped cell's polygon is re-offset to sit along the **group anchor's** face normal instead of its own. Without this, adjacent cells on different faces (cylinder facets, corners) offset in divergent directions, leaving a ~6 mm V-gap at the seam where the face-border shader line on the building underneath becomes visible. Unified offset makes the paint polygons share an exactly coincident edge at seams, hiding the underlying lines. Members at the far end of a 20¬∞ group sit ~2.35 cm above their own face (instead of the usual 2.5 cm) ‚Äî still safely above z-fighting range.

### Suspicious faces (untrustworthy outside direction)

CityGML occasionally ships faces with inverted winding ‚Äî most commonly downward-facing roofs, but also inward-pointing walls on irregular buildings. A one-sided 2.5 cm paint offset along a flipped normal would bury the overlay inside the wall. `makeMeshData` flags a face as `suspicious` when either cue holds:

- **Horizontal-toward-center:** the face's horizontal normal component points toward the mesh bbox center (a correctly-wound exterior wall points *away* from center).
- **Downward-facing:** the face's normal tilts more than 10¬∞ below horizontal (`ny < -sin 10¬∞`). Legitimate exterior surfaces don't face downward at any meaningful angle.

For suspicious faces, both `scanCells` (worker, pre-seeded cells) and `buildCellGeometry` (main thread, runtime-painted cells) emit the polygon offset **on both sides** of the face plane. The paint becomes visible regardless of winding, at the cost of a small amount of extra overdraw limited to the suspicious subset.

### Rendering

For most paint actions, the cell's clipped geometry is already in `cellGeomCache` ‚Äî the worker pre-clipped every cell during the phase-2 seed scan and offset each polygon 2.5 cm along its face's normal. `rebuildBuildingPaint` reads the cache directly and groups cells by colour into one `MeshBasicMaterial` mesh per (building, colour). When a cell isn't cached (rare ‚Äî only happens if `paintStore` has data for a cellKey the worker didn't emit), `buildCellGeometry` clips on the fly using the same Sutherland-Hodgman algorithm with two filters: face normal dot product with the hit normal > 0.7, and triangle centroid within 15 cm of `planeD`.

This approach means paint is bounded by the actual surface geometry ‚Äî edge cells are clipped to the wall boundary rather than hanging over as a full square would.

Per-building paint meshes (one mesh per colour) are tracked in `buildingPaintMeshByBuilding` so `updateCulling()` can toggle them with their parent building when the player moves out of range, and `TileManager._unload` can dispose them in O(1) without scanning the global mesh list.

### Persistence

Paint is stored **per tile**, either on disk at `data/paint/{tileId}.json` via the Vite dev-server middleware (`scripts/tile-paint-plugin.js`) or in Cloudflare R2 behind `worker/index.js` in prod. Both backends speak the same protocol:

- `GET   /paint/:tileId` ‚Üí `{ [cellKey]: cellData }`. Returns an `ETag` header and honors `If-None-Match` (304 on hit ‚Äî used by the pull loop).
- `PATCH /paint/:tileId` ‚Üí body is a **diff**: `{ __ts, __author, [__seed], [cellKey]: cellData | null }`. `__ts` is the client's flush timestamp; `null` entries are erase tombstones (they use `__ts` as their effective `paintedAt`). `__author` is the client's per-device UUID (see *Identity & audit* below). `__seed: true` marks a flush whose dirty cells all originated from procedural seeding ‚Äî exempt from the token bucket. All three envelope fields are stripped before merge and never written into the paint JSON. The server reads the current blob, merges the diff with a `paintedAt` tiebreaker (newer wins), and writes back **under an R2 conditional** (`onlyIf: etagMatches`) ‚Äî retrying up to 5√ó on CAS loss. Two clients painting different cells on the same tile both survive; same-cell races resolve deterministically to the later wall-clock paint.

Stored cell data: `{ color, normal, planeD, paintedAt, authorId }`.

**Identity & audit.** `paintStore` generates a v4 UUID on first run, stores it in `localStorage['graffiti_author_id']`, and stamps it onto every cell it writes (field `authorId`) plus the PATCH envelope (`__author`). The worker strips `__author` out of the diff and ‚Äî if the optional `AUDIT` KV binding is wired up ‚Äî writes one row per `(authorId, ip)` pair with `expirationTtl = 14 days`, refreshed on every PATCH. That gives a 14-day rolling IP‚Üîauthor map for moderation without mixing IPs into the public paint JSON. Cleared `localStorage` yields a fresh identity; stronger identity needs a real login, not in scope.

**Rate limit (token bucket).** Per-authorId bucket backed by the same `AUDIT` KV under `bucket:<authorId>`. Default capacity 500, refill 1 token per 10 s (`BUCKET_CAPACITY` / `BUCKET_REFILL_MS` ‚Äî client mirrors these values until the first header sync, server is authoritative). Each cell write (paint **or** erase) costs 1 token; the seed-complete sentinel and any `__seed: true` batch are free. When a PATCH arrives with cost > available tokens, the worker returns **429** with JSON `{ error:'rate_limited', tokens, refillAt, capacity, refillMs, needed }` and the same state on headers `X-Paint-Tokens` / `X-Paint-Refill-At` / `X-Paint-Capacity` / `X-Paint-Refill-Ms`. Every successful PATCH also echoes those headers so the client stays synced. The client gates paint/erase calls against its local bucket extrapolation, so 429s only happen on multi-tab drift (both tabs optimistically spend their shared budget); when one does fire, the rejected writes become phantom locally until the pull loop reconciles them. The `paint-bank` HUD bottom-right (`index.html#paint-bank`) shows `paint: xx/cap` plus a `+1 in Ys` countdown driven by `paintStore.tickBucket()` at 1 Hz from the render loop. Bucket state inherits a 7-day TTL: inactive users return to a full bucket.

`paintStore.loadTile(tileId, buildingKeys)` is called by `TileManager._doLoad` as soon as meshes are wrapped. It synchronously registers a `buildingKey ‚Üí tileId` mapping, fetches the tile's cells in the background, stashes the returned ETag in `_tileETags[tileId]`, and kicks the pull loop (below). `TileManager._applyCellDataToTile` awaits the fetch before invoking the phase-2 `onTileCellData` callback, so `seedTileCells` runs with server-saved paint already populated.

`paintStore.unloadTile(tileId, buildingKeys)` is called by `TileManager._unload`; it flushes any pending save for the tile, then drops its cells, ETag, dirty map, and in-flight flag.

**Writes** are debounced (`SAVE_DEBOUNCE_MS = 500`) per-tile but tracked at **cell granularity**. Every `paint`/`erase`/`seed`/`markTileSeedComplete` writes to local `cells` *and* to `_dirtyCells: Map<tileId, Map<cellKey, cellData|null>>`. The debounce flush snapshots-and-clears the dirty map, PATCHes the diff, and ‚Äî on success ‚Äî stashes the response ETag; on failure it merges the snapshot back into the current dirty map (newer writes already there win) and re-schedules. `visibilitychange=hidden` triggers an async PATCH; `beforeunload` re-fires each pending PATCH with `keepalive: true` so writes survive page teardown.

**Periodic pull ‚Äî two-tier cadence.** `main.js` recomputes the player's *active tile set* at 1 Hz (standing building tile + standing terrain tile + the tile owning the cell under the crosshair, when pointer-locked) and calls `paintStore.setActiveTiles(...)`. Active tiles poll every `PULL_ACTIVE_MS = 5 s`; every other loaded tile sweeps round-robin on an idle loop at `PULL_IDLE_MS = 20 s` (capped at `PULL_IDLE_MAX_TILES = 20` per tick as a safety rail). The effect: tiles the player is actually interacting with feel near-realtime; farther tiles converge within ~20 s without generating per-tile request traffic. Each pull skips tiles with dirty cells or in-flight saves (and re-checks both flags *after* the fetch awaits, so a paint that lands mid-pull still wins). On a 304 nothing changes; on a 200 `_applyServerState` applies non-dirty cells from the server body (using `paintedAt + color` equality to skip redundant rebuilds), deletes local cells the server no longer has (handles another client's erases), and fires one `subscribeBuildings` callback per touched building ‚Äî which coalesces into at most one `rebuildBuildingPaint` per building per frame via `schedulePaintRebuild`. This is how one client sees another's paint without a full reload.

Seeds are rolled per client at tile load and locked in once seeding finishes. The worker rolls 10% of cells; `seedTileCells` applies them, skipping cells already in `paintStore.cells` (preserves user paint + prior-session seeds). Completion is tracked via a per-tile sentinel (`__seed_complete__`) written to the paint JSON only when `seedTileCells` makes it through every mesh without abandonment ‚Äî see the `abandoned` flag in [main.js#seedTileCells](src/main.js). Tiles with the sentinel skip all future seed rolls (`paintStore.isTileSeedComplete`); tiles without it (first visit, or a prior visit bailed when the player moved out of range mid-seed) get another roll on the next load. The yield between chunks is `requestAnimationFrame`, not `requestIdleCallback`, so seeding makes steady progress under continuous movement instead of starving on idle time. There is also an optional build-time bake (`scripts/bake_seeds.js`, `npm run bake-seeds`) that writes seeds directly to `data/paint/<tileId>.json`; it's not in the default flow because the full city takes ~2 hrs, but it remains the right tool if shared-across-clients starter graffiti is ever wanted.

`paintStore` also maintains a `_byBuilding: Map<"buildingId:meshType", Set<cellKey>>` secondary index. `rebuildBuildingPaint` iterates only that building's cells (via `paintStore.cellsForBuilding(buildingKey)`) instead of scanning the full cells map, so rebuild cost scales with cells-per-building rather than total-painted-cells.

Note: the `planeDKey` in the cell key is just for keying; the exact `planeD` float in cell data is used when reconstructing geometry overlays.

### Terrain paint

Terrain blocks are paintable alongside buildings. Each 2 m √ó 2 m block exposes
up to five paintable faces ‚Äî the tilted top quad and the four cardinal sides
(only when the block stands taller than its neighbour on that side, i.e. the
side is actually emitted by `terrainWorker.buildBlockyTerrainGeometry`). The
cellKey mirrors the building format:

- `buildingId` ‚Äî `terrain_{gx}_{gz}` (one terrain tile = one virtual building)
- `meshType` ‚Äî `top` | `sideN` | `sideS` | `sideE` | `sideW`
- `cellU`, `cellV` ‚Äî the block's in-tile index `ix`, `iz` (both in `[0, res)`)
- `planeDKey` ‚Äî **`iy` for cliff sides, `0` for tops**. A cliff is emitted as
  one quad in the mesh but divided into paint cells along the Y axis at
  `BLOCK_SIZE` (‚âà 1.95 m) intervals ‚Äî matching the horizontal gridlines drawn
  by `gridShader`. `iy = floor(hit.point.y / BLOCK_SIZE)` on a side hit; tops
  aren't subdivided vertically so they stay at `iy = 0`.

`hitCell` recognises terrain hits via `mesh.name === 'terrain'`, recovers
`(ix, iz)` from the hit point + `mesh.userData.bounds`, and picks the meshType
from the face normal (`ny > 0.5` ‚Üí `top`; cardinal sides from `nx`/`nz`). Side
faces sit exactly on cell boundaries in grid space, so the XZ probe point is
nudged half a step inward along the inward face normal before flooring ‚Äî keeps
the index stable against float Œµ at the boundary. No canonicalCellKey pass ‚Äî
terrain keys are unique per face + Y strip.

Terrain skips the `cellGeomCache` entirely. Worker-emitted quads come in a
known 6-vertex-per-face order with constant normals, so `TerrainManager`
scans each tile's geometry once at load time and builds a
`faceStarts: { top, sideN, ..., sideW }` dict of `Int32Array(res*res)`
mapping block index ‚Üí first-vertex offset (or `-1` if the face wasn't
emitted). `buildTerrainCellGeometry` copies those 6 positions out of the
terrain mesh's position buffer and offsets them 2.5 cm along the face
normal. For a side strip it additionally clips the 4 quad corners to the
`[iy¬∑BLOCK_SIZE, (iy+1)¬∑BLOCK_SIZE]` Y band before emitting, so the paint
overlay matches the grid cell the user clicked.

`TerrainManager` registers the 5 buildingKeys in the shared `buildingMeshMap`
at tile load and kicks `paintStore.loadTile("terrain_{gx}_{gz}", keys)`. Once
the fetch resolves, `onTerrainLoaded` rebuilds overlays for any face-type
that has persisted cells. Unload disposes paint overlays via the same
per-buildingKey loop `TileManager._unload` uses, then calls
`paintStore.unloadTile` to flush the dirty save and drop cells from memory.

No seeding on terrain ‚Äî terrain covers ~10√ó the footprint of buildings and
random seed patches would visually dilute rather than densify the graffiti.

### Multiplayer path

Prod runs `worker/index.js` on Cloudflare with an R2 bucket backing it (dev uses the Vite middleware for the same protocol). The cell-diff PATCH + pull loop gives **eventually-consistent multiplayer paint today** ‚Äî within `PULL_INTERVAL_MS` of another client's save, every loaded client re-renders its buildings with the new cells. Push-based sync (WebSocket / Durable Object) would collapse that latency to ~0 and let clients react to individual cell events; `paintStore.subscribe(fn)` and `paintStore.subscribeBuildings(fn)` are already the hooks for it. Tabled until it's worth the paid-plan cost.
